# v0.3.0 - Version Chains and Token Waste

AI Worktree Hygiene now makes versioned work and authored artifacts first-class
checkpoint concerns. The release also sharpens the public positioning: stronger
agents drift faster without phase boundaries, so a phase-clean checkpoint helps
reduce token waste before the next agent builds on a confused branch.

## New Gates

- Version-chain gate: versioned specs, plans, runbooks, branches, worktrees, and
  implementation slices need an explicit version owner, lane, evidence state, and
  closure or continuation decision.
- Implemented-vs-materialized gate: a stale dashboard, report, or generated
  artifact is not proof that source implementation is missing.
- Research gate: provider, memory, code graph, adapter, vendor, and competitor
  research must check prior notes, source/tests, manifests, provenance, and outside
  caches before being called absent.
- Authored-artifact continuity gate: files that affect direction, claims,
  provider boundaries, dashboard behavior, case evidence, or repair decisions must
  link to a parent goal, lane/module, upstream evidence, downstream consumer, claim
  boundary, and next decision.

## Positioning Update

The new public description is:

```text
Phase-clean checkpoint for AI coding agents: stop branch drift, reduce token waste, and keep worktrees, subagent diffs, and versioned slices reviewable.
```

This is intentionally a process claim, not a percentage claim. The project does not
promise measured token savings; it gives agents a stop gate so they spend fewer
cycles carrying unclear branch state forward.

## Verification

`scripts/verify.sh` now requires these v0.3 gates in the skill, release note,
README, launch docs, discoverability notes, and checkpoint template. It also scans
release notes for private project names and internal version labels.
