# v0.2.0 - Phase-Clean and Delegation Gates

AI Worktree Hygiene now checks more than whether tracked source files are dirty.
This release expands the public checkpoint from branch cleanup into phase cleanup:
tracked, untracked, ignored, scratch, generated, and delegated state must all be
explainable before a commit, push, handoff, or next implementation slice.

## New Gates

- `tracked clean` is not `phase clean`; ignored residue must be classified before
  phase switches, handoffs, commits, pushes, or readiness claims.
- Local runtime and generated runtime directories such as `.local/`,
  `node_modules/`, `dist/`, and coverage output need a lane and keep/clean decision.
- Outside-repo scratch outputs need a path, input source, private-data status, and
  cleanup decision.
- Existing-chain audits are required before adding parallel specs, adapters,
  fixtures, dashboard slices, compatibility harnesses, projections, or repair graphs
  in repos that already contain prior evidence.
- Subagent or worker output must pass diff whitelist review before it is accepted.
- Case adapters may read case artifacts, but case sentinel terms and key variants
  must stay out of generic product source, default UI copy, core schemas, and docs.

## Positioning Update

The project is still intentionally small: it is a checkpoint skill, not a workspace
manager, code graph system, or agent platform.

The new short public description is:

```text
Checkpoint skill for AI coding agents: classify dirty worktrees, ignored residue, generated artifacts, subagent diffs, and existing-chain gaps.
```

## Verification

`scripts/verify.sh` now requires this release note and checks that the public docs
describe phase-clean and delegation gates, while continuing to reject private context
leaks in README, docs, examples, templates, and the skill file.
